# GPT공작소 블로그 글작성 실행지시서 v6.2
## 수익형 최신 이슈 + Blogger-safe HTML + 공식근거 강화 기준

이 지시서는 다른 GPT가 블로그 자료를 생성할 때 그대로 복사해서 사용하는 실행 지시서다.  
목표는 단순 정보글이 아니라 **사람들이 최신 사항을 검색할 때 노출될 가능성이 높은 수익형 블로그 글**을 만드는 것이다.

v6.1 보강 사항:
- 본문에 “블로그 운영 관점”, “수익형 글”, “광고 수익” 같은 제작자 관점 표현 금지
- 본문 상단에 최종 업데이트 날짜 표시 필수
- 공식 출처/신청 링크 확인 블록 추가
- `href="#"` 링크를 content.html에 넣는 것 금지
- 공식·공공기관 원출처 우선 확인 강화
- content.html / content.md / content.txt 핵심 내용 일치 강화
- Blogger 업로드 후 표 깨짐 방지를 위해 table 기본 금지 유지

v6.2 보강 사항(Blogger 실제 발행 화면 확인 결과 반영):
- Blogger 테마가 게시글 제목을 자동으로 표시하므로, content.html 본문 최상단에
  동일/유사한 H1 제목을 다시 넣는 것을 금지(제목 중복 방지)
- content.html의 첫 번째 img가 대표(썸네일) 이미지여야 하며, 도입문보다 앞
  (최종 업데이트 날짜 바로 다음)에 배치
- 한 문단은 2~3문장 이하로 작성해 모바일 화면에서 회색 글 덩어리로 보이지
  않게 함
- 오늘 핵심 포스트잇 / 신청 전 확인 카드 / 주의사항 박스 / 공식 링크 박스 등
  Blogger-safe한 박스형 구성 요소를 표준 구조에 포함(style/script 태그 없이
  인라인 스타일만 사용)

# 0. 최우선 목표

블로그 글의 최우선 목적은 **수익성**이다.

단, 본문 독자에게는 “수익성”, “광고”, “블로그 운영” 같은 표현을 절대 노출하지 않는다.  
독자 입장에서는 **최신 정보, 신청 가능 여부, 금액, 절약 효과, 주의사항**을 바로 얻는 글이어야 한다.

반드시 아래 기준을 동시에 만족해야 한다.

1. 최신 이슈 검색 노출 가능성
2. 광고 수익에 유리한 검색 의도
3. 클릭하고 싶은 제목
4. 체류시간을 늘리는 글 구조
5. Blogger 모바일 화면에서 안정적으로 보이는 HTML
6. 이미지 정상 반영
7. 표 깨짐 방지
8. 과장·허위·근거 없는 내용 금지
9. 사용자가 실제로 도움을 받을 수 있는 정보 제공
10. 공식 출처 확인 가능

수익성을 이유로 허위 정보, 확정되지 않은 혜택, 가짜 신청 방법, 조작된 가격, 존재하지 않는 제도, 클릭bait만 있는 글을 만들면 안 된다.

# 1. 오늘 기준 최신 이슈 조사 필수

글을 작성하기 전에 반드시 오늘 기준 최신 이슈를 먼저 확인한다.

다음 분야에서 최소 10개 후보를 조사한다.

- 정부 지원금
- 생활비 절약
- 전기요금 / 가스요금 / 수도요금
- 통신비 / 구독료 / 멤버십 가격 변경
- 금융 / 카드 혜택 / 세금 / 환급
- 자동차세 / 과태료 / 보험료 / 유류비
- 계절 이슈: 폭염, 장마, 한파, 미세먼지, 명절, 휴가철
- 대형 플랫폼 정책 변경: 쿠팡, 네이버, 카카오, 배달앱, OTT
- 소비자 피해 주의보
- 신청 기간이 있는 제도
- 가격 인상·인하·요금제 변경
- 사람들이 당장 검색할 만한 비교형 주제

- 신청 기간

# 2. 주제 선정 기준

10개 후보 중 최종 1개만 선택한다.

## 2-1. 수익성 점수

높은 우선순위:
- 돈을 아끼거나 받는 주제
- 신청·가입·비교·변경·환급·납부 관련 주제
- 사용자가 결정을 앞두고 검색하는 주제
- 광고 클릭 가능성이 있는 주제
- 반복 검색이 가능한 주제
- 최신 변경 사항이 있는 주제
- 공식 신청/확인 페이지가 존재하는 주제

낮은 우선순위:
- 단순 연예 뉴스
- 감상문형 이슈
- 검색 후 바로 이탈할 가능성이 큰 주제
- 정보 수명이 너무 짧은 주제
- 근거가 불명확한 루머성 주제
- 공식 출처를 확인하기 어려운 주제

## 2-2. 검색 의도

아래 의도가 있는 주제를 우선한다.

- 어떻게 신청하지?
- 지금도 받을 수 있나?
- 얼마를 아낄 수 있나?
- 뭐가 바뀌었나?
- 어떤 걸 선택해야 하나?
- 주의할 점은 뭔가?
- 오늘 기준 맞나?
- 공식 신청은 어디서 하지?

## 2-3. 제목화 가능성

제목에 아래 표현이 자연스럽게 들어갈 수 있어야 한다.

- 2026년 최신
- 오늘 기준
- 변경 사항
- 신청 방법
- 놓치면 손해
- 비교 정리
- 주의사항
- 체크리스트
- 지금 확인

단, 과장된 제목은 금지한다.  
예: “무조건 받는”, “전 국민 확정”, “100% 지급” 같은 표현은 실제 근거가 있을 때만 사용한다.

`metadata.json`에는 아래 항목을 포함한다.

```json
{
  "version": "v6.2",
  "type": "blogger_upload_package",
  "topic": "",
  "selected_reason": "",
  "title": "",
  "meta_description": "",
  "main_keyword": "",
  "sub_keywords": [],
  "search_intent": "",
  "target_reader": "",
  "profit_angle": "",

  "fact_check_note": "",
  "official_sources": [],
  "secondary_sources": [],
  "hot_issue_candidates": [],
```

# 5. 제목 작성 규칙

제목은 검색 노출과 클릭을 동시에 고려한다.

## 제목 형식 후보

아래 중 하나를 선택한다.

1. `[오늘 기준] 핵심 키워드 + 변경 사항 + 주의사항`
2. `2026년 핵심 키워드 최신 정리: 신청 방법과 놓치기 쉬운 부분`
3. `핵심 키워드 비교 정리: 지금 선택 전 확인할 점`
4. `핵심 키워드, 올해 달라진 점과 절약 방법`
5. `핵심 키워드 최신 업데이트: 대상, 기간, 방법 한 번에 정리`

## 제목 금지

- 낚시성 제목 금지
- 사실보다 큰 표현 금지
- 확정되지 않은 지급/혜택 단정 금지
- 특수문자 과다 사용 금지
- 너무 긴 제목 금지

# 6. 본문 구성 규칙

## 6-1. 제목 중복 금지 (v6.2 신규)

Blogger 테마는 게시글 제목을 상단에 자동으로 표시한다.  
따라서 **content.html 본문 최상단에 동일하거나 비슷한 제목을 H1(또는 굵은 큰 글씨)로
다시 넣지 않는다.** 그대로 넣으면 실제 발행 화면에서 제목이 두 번 보인다.

- 글 제목은 `metadata.json`의 `title` 필드에서만 관리한다.
- content.html은 제목을 반복하지 않고 **최종 업데이트 날짜부터** 시작한다.
- content.html 안에는 `<h1>` 태그를 쓰지 않는다(본문 소제목은 `<h2>`/`<h3>`만 사용).

## 6-2. 대표 이미지 위치 (v6.2 신규)

Blogger는 본문의 첫 번째 이미지를 대표(썸네일) 이미지 후보로 인식하는 경우가 많다.  
따라서 **content.html에서 가장 먼저 등장하는 `<img>`가 반드시 대표 이미지(썸네일)여야 한다.**

- 대표 이미지는 도입문(여러 문단)보다 앞에 온다.
- 배치 순서: 최종 업데이트 날짜 → **대표 이미지** → 오늘 핵심 포스트잇 → 도입문
- 금지: 도입문 여러 문단을 먼저 쓰고 그 뒤에 이미지를 넣는 구성

## 6-3. 문단 구성 규칙 (v6.2 신규)

모바일 발행 화면에서 긴 문단은 회색 글 덩어리처럼 보여 읽기 어렵다.

- 한 문단은 **2~3문장 이하**로 작성한다.
- 긴 문장은 의미 단위로 나눈다.
- 소제목(h2/h3) 사이 간격을 충분히 둔다(문단을 촘촘히 붙이지 않는다).
- 신청 기준 / 주의사항 / 공식 링크는 일반 문단에 묻히지 않게 **6-4의 박스형
  구성 요소**로 분리한다.

## 6-4. 박스형 구성 요소 (v6.2 신규)

글만 길게 나열하지 말고, 한눈에 파악되는 시각적 구분 요소를 넣는다.  
`style`/`script` 태그는 금지이므로, 아래처럼 **div/p/ul에 인라인 style**만 사용한다
(Blogger에서 깨지지 않는 단순 구조만 사용).

```html
<!-- 오늘 핵심 포스트잇 -->
<div style="background:#fff9db;border:1px solid #f1c40f;border-radius:10px;padding:14px 16px;margin:16px 0;">
  <strong>오늘 핵심</strong><br>
  한 줄로 요약한 오늘의 핵심 내용
</div>

<!-- 신청 전 확인 카드 -->
<div style="border:1px solid #e0e0e0;border-radius:10px;padding:14px 16px;margin:16px 0;">
  <strong>신청 전 확인</strong>
  <ul>
    <li>확인 항목 1</li>
    <li>확인 항목 2</li>
    <li>확인 항목 3</li>
    <li>확인 항목 4</li>
  </ul>
</div>

<!-- 주의사항 박스 -->
<div style="background:#fdecea;border-left:4px solid #d93025;border-radius:6px;padding:14px 16px;margin:16px 0;">
  <strong>⚠️ 주의사항</strong><br>
  실제 확인이 필요한 주의 내용
</div>

<!-- 공식 링크 박스 -->
<div style="background:#eaf1ff;border-radius:10px;padding:14px 16px;margin:16px 0;">
  <strong>공식 확인 · 신청</strong><br>
  <a href="공식URL" target="_blank" rel="noopener">공식 사이트 바로가기</a>
</div>
```

필수 포함 후보(최소 이 중 3개 이상 사용):
- 오늘 핵심 포스트잇
- 신청 전 확인 카드 (또는 먼저 확인할 4가지 체크리스트)
- 주의사항 박스
- 공식 링크 박스

## 6-5. 본문 순서

본문은 반드시 아래 순서로 작성한다. (제목 영역은 Blogger 테마가 자동 표시하므로
content.html에 포함하지 않는다 — 6-1 참고)

1. 최종 업데이트 날짜
2. **대표 이미지**(content.html의 첫 번째 img, 6-2 참고)
3. 오늘 핵심 포스트잇(6-4)
4. 짧은 도입문(2~3문장 단위, 6-3)
5. 신청 전 확인 카드(6-4)
6. 오늘 기준 핵심 요약
7. 공식 확인/신청 링크 박스(6-4)
8. 왜 지금 검색이 늘어나는지
9. 주요 변경 사항 또는 핵심 내용
10. 대상자 / 조건 / 확인 방법
11. 절약·혜택·신청·비교 포인트
12. 주의사항 박스(6-4)
13. 자주 묻는 질문 FAQ
14. 마무리 요약

본문 순수 텍스트 기준 권장 분량:
- 최소 2,500자 이상
- 권장 3,000~4,500자
- FAQ 6~8개
- h2 7~10개 수준

너무 짧은 글은 검색 노출과 체류시간이 약해질 수 있다.  
너무 긴 글은 모바일 이탈이 커질 수 있으므로 카드형/목록형으로 끊어서 작성한다.

# 7. Blogger-safe HTML 규칙

Blogger에 업로드된 뒤 깨지지 않는 것이 중요하다.

## 7-1. 허용 태그

아래 태그 중심으로 작성한다.

- `h2`
- `h3`
- `p`
- `ul`
- `li`
- `strong`
- `em`
- `a`
- `div`
- `img`
- `blockquote`

## 7-2. 금지 태그

아래는 사용하지 않는다.

- `script`
- `style`
- `iframe`
- `form`
- `input`
- `button`
- `table` 단, 정말 필요한 2열 이하 간단 표만 예외 가능
- 복잡한 SVG
- 외부 CSS CDN
- 외부 JS CDN

## 7-3. 표 사용 금지 원칙

Blogger 업로드 이후 표가 깨질 수 있으므로 기본적으로 `table`을 사용하지 않는다.

특히 금지:
- 3열 이상 표
- 4열 표
- 긴 문장이 들어가는 표
- 모바일에서 좌우 스크롤이 필요한 표

표로 쓰고 싶은 내용은 카드형 구조로 변환한다.

### 표 대신 카드형 예시

```html
<div style="border:1px solid #e5e7eb;border-radius:14px;padding:16px;margin:16px 0;background:#ffffff;">
  <h3 style="margin:0 0 10px;">항목 제목</h3>
  <p style="margin:8px 0;"><strong>핵심 내용:</strong> 여기에 핵심 내용을 쓴다.</p>
  <p style="margin:8px 0;"><strong>기대 효과:</strong> 여기에 기대 효과를 쓴다.</p>
  <p style="margin:8px 0;"><strong>주의사항:</strong> 여기에 주의사항을 쓴다.</p>
</div>
```

# 8. 모바일 가독성 규칙

Blogger 모바일 화면을 기준으로 작성한다.

- 한 문단은 2~3문장 이하
- 한 문장은 너무 길게 쓰지 않는다
- 소제목을 자주 넣어 흐름을 끊어준다
- 목록을 적극 사용한다
- 카드형 박스는 너무 많이 남발하지 않는다
- 본문 글씨를 과도하게 키우지 않는다
- Blogger 테마 기본 글씨 크기를 최대한 따른다
- inline style은 간단한 여백, 테두리, 배경 정도만 사용한다
- `font-size` inline style은 가급적 쓰지 않는다
- 카드 안의 h3도 Blogger 기본 크기를 우선한다

# 9. 광고 수익형 글 구조

광고 수익을 위해 체류시간과 스크롤을 유도한다.

단, 본문에는 광고 수익이나 블로그 운영 관점을 쓰지 않는다.  
아래 표현은 content.html, content.md, content.txt 본문에 쓰면 안 된다.

금지 표현:
- 블로그 운영 관점
- 수익형 글
- 광고 수익
- 광고 클릭
- 검색 확장성이 좋습니다
- 체류시간을 늘리기 좋습니다
- 내부링크 전략
- SEO에 유리합니다

이런 내용은 README나 metadata에만 작성한다.

## 9-1. 글 초반

초반 3~5문단 안에 독자가 얻을 이익을 분명하게 말한다.

예:
- “오늘 기준으로 달라진 부분만 먼저 정리합니다.”
- “신청 전 확인해야 할 조건을 먼저 체크하세요.”
- “요금이 달라졌다면 실제 부담이 얼마나 바뀌는지 확인해야 합니다.”

## 9-2. 중간

중간에는 비교, 체크리스트, 주의사항을 배치한다.  
단, 표 대신 카드형 목록으로 작성한다. 신청 전 확인 카드/주의사항 박스는 6-4의
박스형 구성 요소를 사용한다.

## 9-3. 하단

FAQ를 반드시 포함한다.  
FAQ는 검색 노출과 체류시간에 도움이 된다.

FAQ는 6개 이상 작성한다.

# 10. 공식 출처/신청 링크 규칙

최신 정보형 글은 독자가 바로 공식 확인을 할 수 있어야 한다.

본문 중간 또는 하단에 공식 확인 링크 카드 1개를 넣는다.

예시:

```html
<div style="border:1px solid #bfdbfe;border-radius:14px;padding:16px;margin:16px 0;background:#eff6ff;">
  <h3 style="margin:0 0 10px;">공식 안내에서 최종 확인하기</h3>
  <p style="margin:8px 0;">신청 대상, 기간, 금액은 변경될 수 있으므로 발행 전 공식 안내를 한 번 더 확인하세요.</p>
  <p style="margin:8px 0;"><a href="공식URL" target="_blank" rel="noopener">공식 안내 바로가기</a></p>
</div>
```

주의:
- 공식 URL이 확실할 때만 링크를 넣는다.
- 공식 URL이 없으면 링크를 넣지 말고 README에 확인 필요라고 쓴다.
- content.html에 `href="#"` 링크는 금지한다.
- 내부링크 주소가 없으면 content.html에는 링크를 넣지 말고, README에 “발행 전 내부링크 교체 후보”로만 적는다.

# 11. 광고 위치 placeholder 규칙

실제 광고 코드는 넣지 않는다.

광고 위치는 HTML 주석 placeholder로만 표시한다.

```html
<!-- ADSENSE_PLACEHOLDER_TOP -->
<!-- ADSENSE_PLACEHOLDER_MIDDLE -->
<!-- ADSENSE_PLACEHOLDER_BOTTOM -->
```

광고 placeholder는 본문 흐름을 방해하지 않게 배치한다.

권장 위치:
- 도입부 뒤
- 핵심 내용 40~50% 지점
- FAQ 전 또는 마무리 전

# 12. 이미지 규칙

이미지는 총 5장이다.

1. 썸네일 1장
2. 본문 이미지 4장

**v6.2: content.html 안에서 썸네일(`images/thumbnail.png`)이 항상 첫 번째 `<img>`여야
한다. 배치 순서는 6-2/6-5를 따른다 — 최종 업데이트 날짜 바로 다음, 도입문보다 앞.**

## 12-1. 이미지 파일명

- `images/thumbnail.png`
- `images/body-01.png`
- `images/body-02.png`
- `images/body-03.png`
- `images/body-04.png`

## 12-2. 이미지 삽입 HTML

```html
<img src="images/body-01.png" alt="설명 문구" style="max-width:100%;height:auto;border-radius:14px;margin:16px 0;">
```

## 12-3. 이미지 스타일

- 텍스트가 너무 많은 이미지 금지
- 모바일에서 알아보기 쉬운 이미지
- 과장 광고 느낌 금지
- 실제 정보글에 어울리는 이미지
- 저작권 문제가 없는 자체 생성 이미지 사용
- 이미지 안에 금액·기간 등 바뀔 수 있는 정보는 너무 많이 넣지 않는다

# 13. 팩트 체크 규칙

최신 이슈 글은 반드시 사실 확인을 해야 한다.

공식·공공기관·사업 운영기관 원출처를 먼저 확인한다.

우선순위:
1. 정부/지자체/공공기관 공식 페이지
2. 사업 운영기관 공식 페이지
3. 금융사/플랫폼 공식 공지
4. 보도자료
5. 신뢰도 높은 언론 기사
6. 일반 블로그/커뮤니티는 참고용으로만 사용

다음 내용은 확인 없이 단정하면 안 된다.

- 지원금 지급 여부
- 신청 대상
- 신청 기간
- 가격
- 요금
- 정책 변경
- 세금
- 환급
- 법률·제도
- 금융 조건

불확실한 경우 아래 표현을 사용한다.

- “공식 안내 기준으로 확인이 필요합니다.”
- “지역 또는 대상에 따라 달라질 수 있습니다.”
- “신청 전 최신 공지를 확인해야 합니다.”
- “변경될 수 있으므로 최종 제출 전 확인이 필요합니다.”

# 15. 내부 링크 placeholder

내부 링크 주소가 확실하지 않으면 content.html에는 `href="#"`를 넣지 않는다.

대신 README에 아래처럼 적는다.

```text
발행 전 내부링크 교체 후보:
- 관련 절약 방법 정리
- 신청 방법 상세 정리
```

이미 실제 URL이 있을 때만 content.html에 아래처럼 넣는다.

```html
<p>함께 보면 좋은 글: <a href="실제URL" target="_blank" rel="noopener">관련 절약 방법 정리</a></p>
```

# 16. content.html / md / txt 일치 규칙

content.html, content.md, content.txt는 핵심 내용이 서로 달라지면 안 된다.

허용:
- HTML에는 카드형 디자인
- MD/TXT에는 같은 내용을 목록형으로 단순화

금지:
- HTML에만 중요한 주의사항이 있고 MD/TXT에는 없는 경우
- MD/TXT에만 중요한 금액·기간이 있고 HTML에는 없는 경우
- 제목이나 핵심 키워드가 서로 다른 경우

# 18. 생성 전 최종 체크리스트

최종 ZIP 생성 전 아래를 확인한다.

- [ ] 수익성 높은 주제를 골랐는가?
- [ ] 제목에 최신성/검색 의도가 반영됐는가?
- [ ] 본문 상단에 최종 업데이트 날짜가 있는가?
- [ ] 공식 출처가 metadata와 README에 있는가?
- [ ] 필요 시 content.html에 공식 확인 링크가 있는가?
- [ ] 허위·과장 정보가 없는가?
- [ ] Blogger에서 깨질 table 구조를 피했는가?
- [ ] 3열 이상 표가 없는가?
- [ ] 카드형/목록형 중심인가?
- [ ] 모바일에서 읽기 쉬운가?
- [ ] 본문에 “블로그 운영 관점/수익형/광고/SEO” 표현이 없는가?
- [ ] content.html에 `href="#"`가 없는가?
- [ ] 이미지 5장이 포함됐는가?
- [ ] 이미지 경로가 정확한가?
- [ ] style/script 태그가 없는가?
- [ ] 광고 placeholder만 있고 실제 광고 코드는 없는가?
- [ ] metadata.json이 포함됐는가?
- [ ] content.html, md, txt가 모두 있는가?
- [ ] content.html, md, txt의 핵심 내용이 일치하는가?
- [ ] README가 포함됐는가?

v6.2 신규 체크 항목:
- [ ] content.html 본문 최상단에 제목(H1)이 중복되지 않는가?

- [ ] 대표 이미지가 도입문(첫 문단)보다 앞에 있는가?
- [ ] 한 문단이 2~3문장 이하로 나뉘어 있는가?

      3개 이상이 포함되어 있는가?

# 19. 실행용 짧은 프롬프트

다른 GPT에게 바로 붙여넣을 때는 아래 문장을 사용한다.

```
첨부한 GPT공작소 블로그 글작성 실행지시서 v6.2 기준으로 오늘 기준 최신 이슈를 먼저 조사하고, 수익성이 높은 블로그 주제 1개를 선정해 GPT공작소 업로드용 ZIP 패키지를 만들어줘. Blogger-safe HTML 기준을 반드시 지키고, 표는 사용하지 말고 카드형/목록형으로 작성해줘. content.html 본문에는 제목(H1)을 넣지 말고 최종 업데이트 날짜부터 시작해줘. 대표 이미지를 도입문보다 앞(최종 업데이트 날짜 바로 다음)에 배치하고, content.html의 첫 번째 img가 대표 이미지가 되게 해줘. 문단은 2~3문장 이하로 짧게 끊고, 오늘 핵심 포스트잇/신청 전 확인 카드/주의사항 박스/공식 링크 박스를 포함해줘. 실제 이미지 5장도 포함해줘. 본문에는 블로그 운영/수익/광고/SEO 같은 제작자 관점 표현을 넣지 말고, 공식 출처와 최종 업데이트 날짜를 반영해줘.
```
