GPT 공작소 v1.5
====================

신규 모듈 안전 연결 기준 (v1.4부터 유지, v1.5에서도 동일)
- 신규 모듈은 선택 기능입니다.
- 신규 모듈 오류가 기본 업로드/게시판 저장/R2/Blogger 임시저장을 막으면 실패입니다.
- 신규 모듈은 독립 try/catch(safeInit)로 초기화됩니다.
- DOM 요소가 없으면 해당 모듈만 종료됩니다.
- 문제 발생 시 신규 모듈 제거 또는 v1.4/v1.3 안정본 롤백이 가능합니다.

이번 v1.5는 v1.4 대통합 버전의 **안정화 작업**입니다. 새 기능을 크게 추가하지
않고, 실제 모바일 사용 화면과 Blogger 발행 화면에서 확인된 문제를
보정했습니다. 아래 기본 흐름은 v1.4와 100% 동일하게 보호됩니다.

  블로그자료 ZIP 업로드 → 업로드 확인 → 게시판 저장 → 글 선택
  → R2 이미지 업로드 → Blogger 임시저장

====================

PART A. GPT 공작소 모바일 GUI 안정화

A-1/A-2. 게시판 "글 1개 등록" 기준 레이아웃 + 저장 영역 하단 잘림 보정
- 레이아웃 기준을 "게시판이 비어 있을 때"가 아니라 "글이 1개 등록된 상태"로
  다시 잡았습니다. panel--library에 높이 상한(128px)을 둬서 게시판에 글이
  생겨도 블로그 저장 영역이 예측 불가능하게 밀리지 않습니다.
- 블로그 저장 패널 내부 구조를 바꿨습니다: 제목/상태/게시글 상태 표시줄/
  임시저장 시작 버튼/다시 저장 버튼/최종 미리보기·패키지 진단 버튼은
  `save-target-fixed`라는 항상 고정 노출되는 영역에 두고, 저장 진행 이력
  (`save-progress-list`)만 남은 공간에서 내부 스크롤됩니다. 이전에는 이
  영역 전체가 스크롤 컨테이너 안에 있어서, 화면이 좁으면 저장 버튼 자체가
  스크롤해야 보이는 문제가 있었습니다 — 이번에 해결했습니다.
- 블로그 지시서 복사 버튼은 이 구조 밖(패널 맨 아래)에서 항상 고정 노출됩니다.

A-3. Blogger 업로드/임시저장 진행 팝업 고정
- 기존에는 진행 로그가 한 줄씩 늘어날 때마다 팝업 자체 높이가 계속 커졌습니다.
- 이제 팝업은 "현재 단계 요약"(한 줄, 매번 최신 상태로 덮어씀, 팝업 상단
  고정)과 "상세 진행 로그"(고정 높이 160px, 내부 스크롤, 항상 최신 줄이
  보이도록 자동 스크롤)로 나뉩니다. 팝업 자체 크기는 더 이상 커지지 않습니다.
- 완료/실패 상태는 진행 팝업이 끝나면 별도의 명확한 ✅/⚠️ 팝업으로 표시됩니다
  (기존과 동일).

A-4. 백업 팝업 줄바꿈/가독성 보정
- 백업/복원 팝업의 긴 안내문을 2줄 이하로 나누고(`<br/>` 적용), notice-text의
  줄간격(line-height 1.6)과 경고 문구 아래 여백을 넓혔습니다.

A-5. R2 상태 문구 보정
- 기존 "R2 업로드 API(추정)" 같은 개발자식 표현을 없앴습니다.
- 이제 "R2 이미지 업로드"/"Blogger 임시저장" 상태는 "실제 저장 과정에서 최종
  확인됩니다" 형태의 사용자 친화 문구로 안내합니다(설정 → 작업 서버 →
  📡 상세 상태 점검).

A-6. 오류 문구 보정
- 업로드 실패 팝업(업로드 확인 팝업 및 대체 업로드 흐름 모두)에서
  `[upload-module]` 같은 내부 모듈명 접두어를 없애고, "블로그자료 ZIP
  구조가 올바르지 않습니다" + "필수 구성" 목록(metadata.json / content.html
  / images 폴더)으로 안내를 명확히 했습니다.
- 내부 오류 목록(설정 → 오류 목록, ⚠️ 오류 목록 팝업)에는 기존처럼 모듈명이
  남아 있습니다(디버깅 목적).
- 추가 후보 중 "오류 상세 보기에서 원인/조치 표시"는 오류 목록 팝업에 오류
  백과사전 안내 문구를 추가하는 형태로 반영했습니다. "동일 오류 반복 표시
  방지"는 이번 범위에서는 보류했습니다(v1.6 후보, 아래 참고).

A-7. README 문구 정리
- 게시판 검색/필터/정렬 고도화는 **제외 또는 보류**입니다. 이전 버전
  README/완료 보고에 "다음 업데이트 후보"처럼 보이던 표현이 있었다면
  이번 v1.5에서 "보류"로 명확히 정리했습니다. 실제로 게시판 검색/필터/정렬
  기능은 이번에도, 앞으로도 별도 지시가 없는 한 추가하지 않습니다.

A-8. 로고/브랜딩 후보 (v1.6 후보로만 기록, 이번엔 미적용)
- 현재 상단 "🛠️ GPT 공작소"는 그대로 유지했습니다(이번 v1.5는 안정화
  작업이라 화면 자산 변경은 부담이 있다고 판단했습니다).
- v1.6 검토 방향: AI 로봇 + 공구 + 문서 느낌의 작은 아이콘, 모바일 상단에서
  깨지지 않는 단순 SVG/이모지 조합, 애니메이션 없이 로딩 속도에 영향 없는
  가벼운 자산.

PART B. 블로그자료 생성 기준 v6.2 보정

이번 보정은 GPT 공작소 웹 기능 자체가 아니라, **앞으로 생성되는 블로그자료
ZIP의 content.html 품질 기준**을 다룹니다. `data/blog-writing-prompt-v6.1.txt`를
`data/blog-writing-prompt-v6.2.txt`로 교체했고, `js/prompt-copy-module.js`가
이 v6.2 파일을 기본 최신 지시서로 제공하도록 갱신했습니다.

B-1. Blogger 발행 화면 제목 중복 방지
- Blogger 테마가 게시글 제목을 자동으로 표시하므로, content.html 본문
  최상단에 동일/유사한 H1 제목을 다시 넣지 않도록 규칙을 추가했습니다
  (v6.2 지시서 6-1). 글 제목은 metadata.json의 title 필드로만 관리하고,
  본문은 최종 업데이트 날짜부터 시작합니다.

B-2. 대표 썸네일 이미지는 본문 첫 글자보다 위에 배치
- content.html의 첫 번째 `<img>`가 대표(썸네일) 이미지여야 하며, 도입문보다
  앞(최종 업데이트 날짜 다음)에 오도록 규칙을 추가했습니다(v6.2 지시서 6-2).

B-3. 문단 구분 강화
- 한 문단은 2~3문장 이하로 작성하고, 신청 기준/주의사항/공식 링크는 일반
  문단에 묻히지 않게 박스로 분리하도록 규칙을 강화했습니다(v6.2 지시서 6-3).

B-4. 포스트잇/카드/체크리스트형 구성 추가
- 오늘 핵심 포스트잇 / 신청 전 확인 카드 / 주의사항 박스 / 공식 링크 박스를
  표준 본문 순서에 포함했습니다(v6.2 지시서 6-4, 6-5). style/script 태그는
  금지이므로 div/p/ul에 인라인 style만 사용하는 예시 HTML을 지시서에
  포함했습니다.

B-5. 패키지 진단/최종 HTML 미리보기 점검 항목 추가
- `js/package-diagnosis-module.js`에 v6.2 구조 점검 5항목을 추가했습니다:
  1) 본문 최상단 H1 제목 중복 여부
  2) 대표 이미지가 본문 첫 문단보다 위에 있는지
  3) 첫 번째 img가 대표 이미지인지
  4) 긴 문단이 과하게 이어지는지
  5) 핵심 요약/주의/신청/공식 링크가 박스로 구분되어 있는지
- 이 점검은 품질점수나 SEO 점수가 아니라 기존과 동일하게 "정상 / 주의 /
  수정 필요" 3단계로만 표시합니다.
- `js/blogger-final-preview-module.js`의 안내문에 "제목 중복, 대표 이미지
  위치, 문단 길이, 박스형 구성 같은 v6.2 구조 점검은 패키지 진단에서
  확인할 수 있습니다" 문구를 추가했습니다.

신규 모듈 연결 구조 (app-core.js, GptCoreAPI) — v1.4와 동일하게 유지

- app-core.js는 8개 신규 모듈을 각각 독립된 safeInit(try/catch)으로 연결합니다.
- 각 신규 모듈은 필요한 DOM 요소가 없으면 조용히 종료하며, bound 플래그로
  이벤트 중복 연결을 스스로 방지합니다.
- 신규 모듈은 window.GptCoreAPI(선택된 글 조회, 게시판 새로고침, 저장 트리거,
  공용 팝업, 생애주기 이벤트 등록)를 통해서만 core 상태에 접근합니다.
- 생애주기 이벤트(post-selected/upload-confirmed/board-saved/
  board-save-failed/blogger-save-result/data-reset)는 core 처리가 이미
  끝난 뒤에 통지되며, 리스너 하나가 예외를 던져도 다른 리스너나 기본
  흐름에는 영향이 없습니다.
- v1.5에서 이 구조 자체는 수정하지 않았고, jsdom 기반 회귀 테스트로 재검증만
  했습니다(아래 "자동 검증 완료 사항" 참고).
