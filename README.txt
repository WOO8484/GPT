GPT 공작소 v1.8
====================

[v1.8.0 — 카테고리별 TOP1 전체 ZIP 업로드 처리]
기준: GPT공작소 v1.7.0 웹(블로그자료 생성 지시문 v7.0 반영 버전). 이번
업데이트는 지시문 수정이 아니라 웹 프로그램 업데이트입니다. 목표는 "전체
묶음 ZIP"을 업로드했을 때 내부의 카테고리별 TOP1 개별 ZIP 9개를 자동
인식해 게시판 글 9개로 등록하는 것입니다.
- 웹이 인식하는 전체 묶음 ZIP 구조:
  category_top1_zips/01_오늘의핫이슈_TOP1.zip ~ 09_지역생활정보_TOP1.zip
  (파일명 규칙만 맞으면 폴더 위치는 유연하게 인식합니다)
- upload-module.js: 내부 파싱 로직을 scanEntries() 순수 함수로 분리했습니다
  (기존 단일 ZIP 업로드 동작은 완전히 동일 — 로직을 옮기기만 했습니다).
  0N_..._TOP1.zip 파일을 발견하면 전체 묶음 ZIP으로 인식해, 내부의 개별
  ZIP 9개를 각각 다시 열어(같은 scanEntries 재사용) 카테고리별로 파싱합니다.
  일부 카테고리가 손상되어도 나머지 카테고리는 계속 처리합니다.
- upload-confirm-module.js: 전체 묶음 ZIP이면 카테고리 9개 인식 결과(성공/
  실패, 제목)를 목록으로 보여주고, 저장 버튼이 "게시판에 N개 저장"으로
  바뀝니다. 개별 ZIP 9개를 각각 게시판 글로 저장하며, 전체 묶음 ZIP
  자체나 TOP5 후보 파일은 게시글로 저장하지 않습니다.
- app-core.js: 카테고리 개수만큼 기존 LibraryModule.savePost()를 반복
  호출합니다(저장 로직 자체는 무수정). 게시판 카드에 카테고리 배지를
  추가했습니다.
- naver-copy-module.js: 네이버 태그 표시/복사 우선순위를 naver_tags.txt
  우선으로 변경했습니다(이전 라운드는 metadata.naver_tags 우선).
- 신규 topic-detail-module.js(선택 기능): 선택한 글의 top5/0N_candidate.md
  5개, top5_summary.md, selected_topic.md 원문을 읽기 전용 팝업으로
  보여줍니다. 저장/발행/재선정 동작은 없습니다.
- 자체 검증: Node 샌드박스에서 실제 upload-module.js를 그대로 불러와
  (1) 9개 카테고리 목업 전체 묶음 ZIP 파싱 테스트, (2) 기존 단일 ZIP
  회귀 테스트를 모두 통과 확인(제목/카테고리명/이미지 5장/TOP5 5개/
  selected_topic.md/naver_tags.txt 특수문자 검사까지 정상 동작).
- 로그인/Worker 인증/R2 업로드/Blogger 저장/기존 단일 ZIP 업로드 흐름은
  전혀 건드리지 않았습니다(diff 동일 확인).
자세한 내역은 완료 보고를 참고하세요.

1. 버전 표기 전체 수정 (버그 수정 포함)

v1.5 산출물을 다시 확인한 결과, 로그인 화면과 설정창의 버전 표기가 실제로는
갱신되지 않고 있었습니다(로그인 화면 "v1.4", 설정창 현재 버전 "v1.4", 기준
버전 "v1.3"로 남아 있었음 — v1.5 작업 당시 이 두 위치를 놓친 실수였습니다).
이번 v1.6에서 아래처럼 전부 바로잡았습니다.

- 메인 로그인 화면 버전(login-version): v1.6
- 설정창 "버전 정보" 현재 버전: GPT공작소 v1.6
- 설정창 "버전 정보" 기준 버전: GPT공작소 v1.5
- version.json displayVersion: v1.6
- README.txt: v1.6 기준 전면 갱신
- 별도 변경 요약 파일도 v1.6 기준으로 작성

화면(사용자가 보는 UI 텍스트) 어디에도 v1.3/v1.4 표기가 남아있지 않음을
전수 검색으로 확인했습니다(완료 보고 참고). 코드 주석에 남아있는 "v1.3:",
"v1.4:" 같은 표기는 그 기능이 어느 버전에서 추가됐는지 기록하는 개발용
히스토리 주석이며, 화면에는 노출되지 않습니다.

2. 오류 목록 문구 실제 수정

설정창 오류 목록 카드와 ⚠️ 오류 목록 팝업 모두에서 "[upload-module] 필수
파일 누락" 같은 개발자용 문구가 그대로 보이지 않도록, 화면에 표시하는
시점에 사용자 친화 문구로 변환하는 함수(friendlyErrorTitle)를
app-core.js와 settings-module.js 양쪽에 추가했습니다.

- 변환 전(저장된 원본, 안 바뀜): [upload-module] 필수 파일 누락
- 변환 후(화면 표시): 블로그자료 ZIP 구조 오류

이 변환은 "표시할 때"만 적용되므로, localStorage에 예전 방식으로 이미
저장되어 있던 오류 기록도 다시 열면 자동으로 친화 문구로 보입니다. 오류의
detail(상세 사유)은 그대로 함께 표시됩니다. 업로드 실패 팝업(업로드 확인
팝업, 대체 업로드 흐름)은 v1.5에서 이미 아래 문구로 개선되어 있었고, 이번에
다시 확인했습니다.

  블로그자료 ZIP 구조가 올바르지 않습니다.
  프로그램 ZIP, Claude 작업 ZIP, 일반 압축파일은 업로드 대상이 아닙니다.
  GPT 공작소 블로그자료 ZIP을 업로드하세요.

  필수 구성:
  - metadata.json
  - content.html
  - images 폴더

3. 상단 로고/이모티콘 실제 적용

앱 헤더(로그인 후 메인 화면 상단)의 제목을 "🛠️ GPT 공작소"에서
"🤖 GPT 공작소"로 실제로 교체했습니다(index.html의 .app-header__title).
공구 이모지 단독 표시는 더 이상 쓰지 않습니다. 로그인 화면의 "🔐 GPT
공작소"는 자물쇠 아이콘이 암호 입력 화면이라는 의미를 갖고 있어 그대로
유지했습니다(브랜드 로고가 아니라 화면 성격을 나타내는 아이콘).

4. 블로그 지시서 복사 버튼 위치 변경

블로그 지시서 복사 버튼을 블로그 저장 카드(panel--save) 내부에서 완전히
꺼내, app-container 최하단의 독립 영역(.bottom-actions)으로 옮겼습니다.
저장 버튼/최종 미리보기/패키지 진단 버튼과 시각적으로 섞이지 않고, 상단에
구분선(border-top)이 있는 별도 바 형태로 항상 화면 맨 아래 고정됩니다.
prompt-copy-module.js는 버튼 id만 그대로 참조하므로 이 이동으로 인한 JS
쪽 코드 변경은 없습니다.

5. 게시판 레이아웃 보정 (글 없음 / 글 있음)

- 글 없음: "저장된 글이 없습니다…" 안내 문구가 게시판 영역 안에서 flex로
  중앙 정렬됩니다. 기존에는 고정 padding(40px)이 panel--library의 높이
  상한(128px)보다 커서 잘릴 수 있었는데, 이번에 flex:1 + 중앙 정렬 방식으로
  바꿔 어떤 높이에서도 잘리지 않고 중앙에 표시됩니다.
- 글 있음: 카드 제목은 기존처럼 1줄 말줄임 처리되고, 카드 메타(날짜+상태
  배지) 줄에도 동일하게 방어적 말줄임을 추가했습니다. 기준은 "빈 게시판"이
  아니라 "글 1개 등록된 상태"입니다.

6. 블로그 저장 카드창 고정

블로그 지시서 복사 버튼이 카드 밖으로 빠지면서 저장 카드 내부에 여유 공간이
생겼고, min-height를 260px→280px로 올렸습니다. 저장 카드 내부 구조는
v1.5부터 이어진 대로 "고정 영역(제목/상태/게시글 상태줄/저장 버튼/다시
저장 버튼/최종 미리보기·패키지 진단 버튼)" + "내부 스크롤 영역(저장 진행
이력)"으로 분리되어 있어, 저장 버튼/최종 미리보기 버튼/패키지 진단 버튼이
항상 온전히 보이고 하단이 잘리지 않습니다. 전체 페이지는 여전히 스크롤되지
않고, 필요한 영역(게시판 목록, 저장 진행 이력 등)만 내부 스크롤됩니다.

7. 블로그 지시서 복사 팝업 고정

이 팝업에 popup-panel--fixed 클래스를 적용해 높이를 420px로 고정했습니다
(내부 스크롤도 비활성화: popup-panel__scroll--noscroll). [전체 복사] 클릭
후 "복사 완료" 문구가 나타나도 그 문구 영역(prompt-copy-result)이 항상
최소 높이를 미리 차지하고 있어서 팝업 자체 크기가 커지지 않습니다. 저장
확인 팝업에도 같은 스크롤 비활성화 클래스를 적용했습니다(이 팝업은 내용이
항상 고정이라 원래도 크기가 변하지 않았지만, 명시적으로 스크롤을 막아
확실히 했습니다).

내부 스크롤이 계속 허용되는 팝업: 진행 로그 팝업(상세 로그 부분만),
⚠️ 오류 목록 팝업, 📚 전체 게시판 팝업, 🔍 최종 미리보기/🩺 패키지 진단
같은 긴 미리보기 팝업.

8. 블로그 저장 진행 팝업 고정 (v1.5에서 이미 구현, 이번에 재확인)

"현재 단계 요약"(상단 고정 한 줄)과 "상세 로그"(고정 높이 160px, 내부
스크롤)로 나뉘어 있어 진행 텍스트가 늘어나도 팝업 자체 크기는 변하지
않습니다. 완료/실패는 별도의 명확한 ✅/⚠️ 팝업으로 표시됩니다.
